# 本地推理与可视化说明

本文档说明如何在本地对 WiFiPose 数据执行推理，并把结果整理成一套可持续复用的可视化流程。

当前默认可视化模式已经调整为 `raw`：

- `raw`：直接画模型原始预测，适合看当前模型真实输出能力
- `matched`：画与 GT 贪心匹配后的预测，适合做误差分析
- `both`：两种图都输出

同时，当前默认只显示分数最高的 `top3` 预测，避免 100 个 query 全部画出来导致图面不可读。

当前流程已经支持：

1. 单样本推理与可视化
2. 连续多样本批量推理
3. 指定多个样本索引推理
4. 每个样本单独落盘
5. 自动生成整批汇总 `summary.json`

## 一、先理解这套可视化在看什么

这套可视化看的不是普通图像输入输出，而是：

- 输入：WiFi CSI 处理后的张量
- 输出：人体 3D 关键点坐标和分数

也就是说，`png` 只是为了让人能快速肉眼检查结果而额外生成的可视化文件，不是模型原生输出。

模型原生输出更接近这些文件：

- `*_pred.npy`
- `*_pred_matched.npy`
- `*_metrics.json`

其中：

- `pred.npy` 更适合看模型的真实原始能力
- `pred_matched.npy` 更适合做误差分析

## 二、模型输入是什么

每个样本主要有两部分原始数据：

- `csi/<sample>.mat`
- `keypoint/<sample>.npy`

数据集读取逻辑在：

```bash
opera/datasets/wifi_pose.py
```

处理流程大致是：

1. 从 `.mat` 中读取 `csi_out`
2. 还原为复数 CSI
3. 对幅度做 `dwt_amp`
4. 对相位做 `phase_deno`
5. 将幅度特征和相位特征拼接
6. 最终整理成一个 WiFi 特征张量送进模型

按当前实现，可以把单样本输入近似理解为：

```text
3 x 3 x 20 x 60
```

再带上 batch 维度后，模型入口看到的张量形状近似是：

```text
bs x 3 x 3 x 20 x 60
```

这里不是 RGB 图像，也不是视频帧，而是 CSI 特征张量。

## 三、模型内部是怎么吃这个输入的

模型主体配置在：

```bash
configs/wifi/petr_wifi.py
```

模型入口在：

```bash
opera/models/detectors/petr.py
```

这份实现没有走标准的图像 ResNet 特征提取路径，而是直接：

1. 读入 `img`
2. 将最后一个维度为 `60` 的 WiFi 特征展开成 token 序列
3. 经过一个 `Linear(60, 256)`
4. 再送入 `PETRHead`

可以把它理解为：

- 每个 token 原始维度是 `60`
- 先映射到 `256`
- 再用 PETR 的 query 机制做人体关键点预测

## 四、模型输出是什么

当前关键配置是：

- `num_query=100`
- `num_keypoints=14`
- `test_cfg.max_per_img=100`

含义是：

- 模型内部固定有 `100` 个 query
- 每个 query 都会尝试预测一个候选人体
- 推理阶段最多保留 `100` 个候选结果

所以你看到的“100 个结果”不是因为样本里真的有 100 个人，而是因为：

- 模型天然会吐出最多 100 个候选
- 其中大部分通常只是低分候选

单个候选结果主要包括：

- 一个分数
- 一组 `14 x 3` 的人体关键点

这里的 `3` 表示三维坐标轴，不是 RGB 通道。

## 五、为什么默认只看 top3

如果把全部 `100` 个 query 的预测都画出来，图会非常乱，几乎没有阅读价值。

因此现在脚本默认行为是：

1. 先按 `score_thr` 过滤
2. 再按分数从高到低排序
3. 只保留 `top3`

这样更符合“快速判断当前模型能力”的目标。

当前默认就是：

```bash
--top-k 3
```

如果你要调整：

- `--top-k 1`：只看最强一个
- `--top-k 5`：看前五个
- `--top-k 10`：看前十个
- `--top-k 0`：保留全部过滤后的预测

## 六、raw 和 matched 的区别

这两个模式用途不同：

- `raw`
  - 直接画模型原始预测
  - 适合看模型当前真实输出能力
- `matched`
  - 将预测和 GT 做贪心匹配后再画
  - 适合做误差分析
- `both`
  - 两种图都保存

如果你的问题是“这个模型现在到底能不能报出合理的人体姿态”，应该优先看：

```bash
--vis-mode raw
```

如果你的问题是“它和 GT 差了多少”，再看：

```bash
--vis-mode matched
```

## 八、脚本会做什么

脚本会执行以下流程：

1. 读取配置和 checkpoint
2. 构建 `WifiPoseDataset`
3. 对指定样本逐个执行推理
4. 读取 GT 关键点
5. 将预测人与 GT 做贪心匹配
6. 计算单样本 MPJPE
7. 生成 GT / Pred 并排的 3D 骨架图
8. 为每个样本保存结果
9. 对整批结果生成 `summary.json`

## 九、输出目录结构

假设输出目录为：

```bash
/Volumes/Work/Person-in-WiFi-3D-repo/outputs/infer_vis/batch_test
```

则输出结构类似：

```text
batch_test/
  summary.json
  S11_01_308/
    S11_01_308_gt.npy
    S11_01_308_pred_matched.npy
    S11_01_308_metrics.json
    S11_01_308_vis_raw.png
  S11_01_309/
    ...
```

如果加了 `--save-raw-pred`，每个样本目录下还会额外保存：

```text
*_pred.npy
```

其中：

- `*_gt.npy`：GT 关键点
- `*_pred.npy`：模型原始预测
- `*_pred_matched.npy`：按 GT 顺序重排后的预测
- `*_metrics.json`：单样本指标与匹配详情
- `*_vis_raw.png`：GT / 原始预测 的 3D 骨架对比图
- `*_vis_matched.png`：GT / 匹配预测 的 3D 骨架对比图
- `summary.json`：整批样本的汇总结果

## 十二、如何看输出结果

最关键的是先看下面两个文件：

- `*_vis_raw.png`
- `*_metrics.json`

其中：

- `vis_raw.png` 用来肉眼看模型当前预测能力
- `metrics.json` 用来看分数、保留人数、匹配误差

如果你打开 `metrics.json`，重点关注这些字段：

- `num_pred_persons`
- `prediction_scores`
- `top_k`
- `vis_mode`
- `metrics_mm.mpjpe`

例如：

- `num_pred_persons=3` 说明当前图中只保留了 top3
- `prediction_scores` 可以直接看到这 3 个预测的分数
- `mpjpe` 可以看到匹配后误差有多大

每次批量运行结束后，输出目录下还会生成：

```bash
summary.json
```

这个文件包含：

- 推理所用 checkpoint
- 数据集目录
- 本次运行参数
- 样本级指标列表
- 整批样本的聚合指标

聚合指标目前包含：

- `num_samples`
- `num_samples_with_match`
- `mean_mpjpe`
- `median_mpjpe`
- `min_mpjpe`
- `max_mpjpe`

样本列表默认按 `sample_index` 排序，也可以改成按 `mpjpe` 排序：

```bash
python /Volumes/Work/Person-in-WiFi-3D-repo/tools/wifi_pose_infer_vis.py \
  --config /Volumes/Work/Person-in-WiFi-3D-repo/configs/wifi/petr_wifi_remote.py \
  --checkpoint /Volumes/Work/Person-in-WiFi-3D-repo/checkpoints/best_mpjpe_epoch_2_20260406.pth \
  --dataset-root /Volumes/Work/wifipose_dataset/test_data \
  --mode test \
  --sample-index 0 \
  --max-samples 20 \
  --summary-sort-key mpjpe \
  --summary-sort-desc \
  --device cpu \
  --output-dir /Volumes/Work/Person-in-WiFi-3D-repo/outputs/infer_vis/batch_ranked
```

这样更适合先看误差大的样本。

## 十三、推荐使用方式

当前建议把本地可视化流程分成三步：

1. 先跑 `5` 到 `10` 个样本，确认模型输出形态正常
2. 再批量跑一段连续区间，检查 `summary.json` 的整体分布
3. 最后针对误差大的样本单独复查 `vis_raw.png` 或 `vis_matched.png`

这个流程比一上来直接跑全量更稳，定位问题也更快。
