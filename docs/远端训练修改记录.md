# 远端训练修改记录

## 文档目的

本文档记录本次为 `Person-in-WiFi-3D-repo` 打通 DLP 远端训练链路所做的代码修改、脚本补充、兼容性修复与验证结果。

目标远端目录约定如下：

- 远端仓库目录：`/cloud/tmp/yc/Person-in-WiFi-3D-repo`
- 远端数据目录：`/cloud/tmp/yc/Person-in-WiFi-3D-repo/data/wifipose`

## 新增文件

### `remote_platform/dlp/dlpflow_train.py`

新增 DLP 训练任务提交入口。

主要职责：

- 使用 `JobFlow` 生成并提交远端训练任务
- 将 NFS 根目录挂载到远端容器
- 通过环境变量传递远端仓库路径、数据路径、工作目录、GPU 配置和额外 `cfg-options`
- 统一以 `bash remote_platform/dlp/run_remote_train.sh` 作为远端训练入口

### `remote_platform/dlp/run_remote_train.sh`

新增远端训练主脚本。

主要职责：

- 导出远端代理环境变量
- 设置 `PYTHONPATH`、`TORCH_HOME`、`TORCH_EXTENSIONS_DIR`
- 清理数据目录中的 macOS 元数据文件 `._*`
- 按需安装远端依赖与本地 editable 包
- 在训练前验证数据目录结构
- 使用 `tools/train.py` 启动训练
- 支持单进程训练与分布式训练两种模式

### `remote_platform/dlp/validate_remote_dataset.py`

新增远端数据校验脚本。

主要职责：

- 检查训练集和测试集的目录是否齐全
- 检查 list 文件是否存在
- 统计 `.mat` 和 `.npy` 文件数量，忽略 `._*`
- 在数据为空或缺失时提前失败，避免任务进入训练后才报错

### `remote_platform/dlp/manifests/train_5samples.txt`

新增 5 样本训练测试清单。

作用：

- 在不改真实数据根目录的前提下，只限制训练样本数量
- 通过 `data.train.list_file=...` 的方式做 5 样本冒烟测试
- 避免用软链接或伪造数据根目录

## 修改文件

### `opera/datasets/wifi_pose.py`

本文件做了两类关键修改。

#### 1. 支持可选的 `list_file`

在 `WifiPoseDataset` 中新增了可选参数 `list_file`。

目的：

- 允许训练时继续使用真实数据目录
- 通过传入子集清单限制训练样本
- 满足“只抽 5 个文件训练测试”的需求

#### 2. 修复 `h5py 3.x` 兼容性

原始代码：

```python
h5py.File(csi_path)['csi_out'].value
```

已修改为：

```python
with h5py.File(csi_path, 'r') as csi_file:
    csi = csi_file['csi_out'][()]
```

原因：

- 远端环境中的 `h5py` 为 3.x
- `.value` 接口已不可用
- 原代码会在训练读第一批数据时直接报错

### `third_party/mmcv/setup.py`

将 MMCV 扩展编译参数从 C++14 调整为 C++17。

原因：

- 远端 PyTorch 编译环境要求 C++17
- 原始版本在远端安装 `mmcv` 时会报：
  - `You need C++17 to compile PyTorch`
  - `C++17 or later compatible compiler is required`

### `third_party/mmcv/mmcv/utils/config.py`

为 `yapf.FormatCode` 增加兼容回退逻辑。

修改逻辑：

- 先尝试 `FormatCode(..., verify=True)`
- 若 `verify` 参数不被当前 `yapf` 支持，则自动退回到 `FormatCode(...)`

原因：

- 新版本 `yapf` 移除了 `verify` 参数
- 原始 MMCV 在 `cfg.dump(...)` 阶段就会失败，训练根本起不来

### `third_party/mmcv/mmcv/parallel/_functions.py`

补充了 `torch` 新版本下的设备类型兼容处理。

修改逻辑：

- 在调用 `_get_stream()` 前，将整数 GPU id 显式转换为 `torch.device('cuda', id)`

原因：

- 当前远端 `torch` 在该路径上要求的是 `torch.device`
- 原实现直接传 `int`，导致单卡训练在 `MMDataParallel` 的 scatter 阶段报：
  `AttributeError: 'int' object has no attribute 'type'`

### `remote_platform/dlp/requirements-remote-base.txt`

将 `yapf` 固定为：

```txt
yapf<0.40
```

原因：

- 防止远端安装与 MMCV 不兼容的高版本 `yapf`
- 与 `mmcv/utils/config.py` 的兼容回退配套使用

### `remote_platform/dlp/run_remote_train.sh`

除主体逻辑外，还做了两个关键修复。

#### 1. 增加代理环境变量

加入：

```bash
export http_proxy="http://p.truesightai.com:1087"
export https_proxy="http://p.truesightai.com:1087"
```

原因：

- 远端安装依赖与下载预训练权重需要走代理

#### 2. 修复 zsh 与 bash 语法不兼容

原先使用了 zsh 风格的变量展开，远端用 `bash` 启动后会报：

```bash
bad substitution
```

现改为 bash 兼容写法：

```bash
read -r -a EXTRA_CFG_OPTIONS <<< "${PIWIFI_CFG_OPTIONS}"
```

## 本次处理过的问题与对应修复

### 1. MMCV 编译失败

现象：

- 远端安装 `mmcv` 时因 C++ 标准过低失败

修复：

- 把编译标准提升到 C++17

### 2. 5 样本测试无法被正确识别

现象：

- 用伪造子集目录或软链接方案时，数据校验不过

修复：

- 保持真实数据根目录不变
- 用 `list_file` 方式限制训练样本

### 3. 训练脚本 bash 不兼容

现象：

- 远端脚本启动即报 `bad substitution`

修复：

- 替换为 bash 兼容写法

### 4. 远端代码不完整导致导入失败

现象：

- `opera.models.utils` 导入失败

修复：

- 将缺失目录重新同步到远端仓库

### 5. MMCV 配置导出阶段失败

现象：

- `FormatCode() got an unexpected keyword argument 'verify'`

修复：

- 固定 `yapf<0.40`
- 同时给 MMCV 增加回退逻辑

### 6. H5PY 接口兼容失败

现象：

- 第一批数据读取时报：
  `AttributeError: 'Dataset' object has no attribute 'value'`

修复：

- 将 `.value` 改为 `[()]`

### 7. MMDataParallel scatter 兼容失败

现象：

- 第一轮 `train_step` 时报：
  `AttributeError: 'int' object has no attribute 'type'`

修复：

- 在 scatter 前将整数 GPU id 显式转为 `torch.device`
